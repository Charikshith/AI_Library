# Curation rules for AI Library

This repo is a curated catalogue of open-source AI tools & resources. The owner adds
tools by saying **`add <tool name or URL>`**. Follow these rules exactly so the library
stays consistent across sessions.

## Source of truth

The site has two content areas, both markdown-sourced and both generated by
`scripts/build.py`. **Never hand-edit the generated `.html` files** — edit the
markdown source and rebuild.

- **Tools catalogue:** `README.md` is the source of truth. All tool data lives in the
  markdown tables between the `<!-- TOOLS:START -->` and `<!-- TOOLS:END -->` markers.
  Generated output: `index.html` (repo root).
- **Notes:** source markdown lives in `notes/<subject>/markdown/` (see the Notes section
  below). Generated output: `notes.html` (repo root), plus each subject's pages in
  `notes/<subject>/html/` (`index.html` + one file per chapter).
- **Per subject, source and output are split into `markdown/` and `html/` subfolders** so
  both sit together but stay tidy. The top-level `index.html` and `notes.html` are written
  to the repo root, which is where **GitHub Pages serves from** (main branch root).
- `scripts/build.py` parses both sources and regenerates all HTML. The two areas are linked
  by the top nav (Library · Notes).

## Adding a tool — step by step

1. **Dedupe.** Search the README for the tool first. If it already exists, update the
   existing row instead of adding a duplicate.
2. **Confirm what it is.** If unsure, look it up (WebSearch/WebFetch) so the description
   and link are accurate — this repo is public.
3. **Description:** one concise, neutral sentence on what it does. No marketing language.
   First-person notes ("why I saved it") are not used. Inline `code` is fine.
4. **Category:** reuse an existing `##` category if one fits. Otherwise create a new one
   (see below). One-offs that don't fit go under **Misc** until ~3 accumulate, then
   promote them to their own category.
5. **Tags:** 2–5 lowercase tags, no `#` inside the backticks is fine but render them as
   `` `#tag` ``. Reuse existing tags before inventing new ones (check the README first).
6. **Date:** stamp with today's date in `YYYY-MM-DD` format.
7. **Insert newest-first:** add the new row directly under the table's header/separator
   (top of its category), so recent finds surface first.
8. **Maintain the Table of Contents:** if you created a category, add a link to it,
   keeping the ToC ordered **alphabetically**. The anchor is the heading lowercased with
   spaces → `-` and `&`/punctuation removed (e.g. `## AI Agents & Skills` → `#ai-agents--skills`).
9. **Build:** run `python scripts/build.py` (falls back to `py scripts/build.py` on Windows
   if `python` isn't found). Confirm it reports the expected tool count.
10. **Commit** `README.md` and the regenerated HTML (`index.html`, and any changed
    `notes/*/html/` pages) together with a message like `add: <tool name>`. Only push
    when the owner asks.

## Row format

```markdown
| [Tool Name](https://link) | One-line description. | `#tag1` `#tag2` | YYYY-MM-DD |
```

## Categories

- Headings are `## Title Case`, ordered alphabetically in the Table of Contents.
- Within a category table, rows are ordered newest-first (by date added).
- Keep category names broad enough to grow (e.g. "AI Agents & Skills", not "Claude Skills").

## Notes — subjects & chapters

Notes are long-form study material organised as **Subjects → Chapters**.

- A **subject** is a folder: `notes/<subject-slug>/` (e.g. `notes/mastering-python/`).
- Inside it: `_subject.json` (metadata), a `markdown/` subfolder (source chapters), and a
  generated `html/` subfolder (do not hand-edit). The build creates `html/`.
- Each subject has a `_subject.json` with `{ "title", "description", "tags": [...], "added" }`.
- Each **chapter** is a markdown file in `markdown/`. **Order and chapter number come
  from a numeric filename prefix** (e.g. `02 - Introduction to Python.md`). The chapter
  title is the filename with the leading `NN - ` and extension stripped.
- Chapters are rendered client-side (marked.js + highlight.js gruvbox theme + mermaid.js)
  so code blocks, tables, headings, and ```mermaid diagrams all render on-theme. Content is
  embedded inline. Chapter ordering is **numeric** by filename prefix (so `2` sorts before
  `10`); unnumbered files (e.g. `Resources.md`) sort last.

**Adding a subject:** create `notes/<slug>/` with a `_subject.json` and a `markdown/`
subfolder, drop the numbered chapter `.md` files into `markdown/`, then run the build
(it creates `notes/<slug>/html/`).

**Adding a chapter:** drop a numbered `.md` into the subject's `markdown/` folder and
rebuild. To keep ordering clean, zero-pad numbers (`02`, not `2`).

**Style:** both the Notes index (subjects) and subject pages (chapters) use the flat
"editorial list" — numbered rows with hairline rules, no boxes. Keep it that way.

## Design / landing page

- The HTML template (structure + layout) lives inside `scripts/build.py`. Edit it there,
  never in the generated `.html`.
- Data is embedded inline in the generated HTML so pages work both locally and on Pages.

### Themes

- **Themes live in `scripts/themes.py`** as a `THEMES` registry of design tokens (palette,
  fonts, grain, code + diagram colours). The structural CSS in `build.py` references CSS
  variables that each theme supplies — themes change only the look, not the layout.
- Build with a theme: `python scripts/build.py --theme <name>` (default: `paper`; also
  settable via `AI_LIBRARY_THEME`). List them: `python scripts/build.py --theme ?`.
- Current themes: **paper** (default — warm "archive / reading-room": ivory bg, Fraunces
  serif, rust accent), **ink** (dark, amber accent), **slate** (cool light, indigo accent).
- To add a theme: copy a block in `themes.py`, tweak the tokens, rebuild with `--theme`.
  Leave `paper` as the committed default unless the owner picks another.
